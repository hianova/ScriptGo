# ScriptGo 專案原始碼深度審計報告

## 一、 專案概述與 README 聲稱分析
`ScriptGo` 是一套專為高頻交易（HFT）超低延遲設計的、基於暫存器的虛擬機器。其 README 聲稱實現了航天級優化、完全零分配（Zero-Allocation）、嚴格 `#![no_std]` 兼容，並在背景腳本熱更新時達到 250ns 的極限延遲，保證 0% 的丟包率（Drop Rate）。

### 1.1 聲稱合理性評估
代碼審計確認，該虛擬機的核心執行引腳（暫存器數組、調用棧等）確實採用了預分配的靜態數組，避開了運行時的堆分配，在運算時能滿足超低延遲的要求。但對於「250ns 熱更新且 0% 丟包率」的極限聲稱，其代碼中缺乏相應的高精度同步保證，多個多線程場景下的內存同步機制不夠嚴密。

對於高頻交易系統而言，250ns 的熱更新時間需要有極其穩定的處理器亂序執行（Out-of-Order Execution）防護。如果在熱加載時前台交易線程仍在全力執行字節碼，任何不當的指令寫入都會導致 CPU 指令快取（I-Cache）失效或流水線清空（Pipeline Flush），引發高達數微秒的停頓，這與 250ns 的宣稱存在衝突。

## 二、 功能完備性與妥協模擬審查
`ScriptGo` 實現了完整的固定長度指令集解碼與虛擬機執行狀態機。

### 2.1 熱更新與併發同步的漏洞
README 中提到的「Chaos Monkey Hot-Reload」測試，在 `benches/concurrency_bench.rs` 中被作為性能測試：
```rust
let writer_handle = thread::spawn(move || {
    // background write and parse
});
```
然而，虛擬機在熱更新時，背景執行緒會動態覆寫指令區，而前台執行緒在沒有使用足夠強度的內存屏障（Memory Barrier）或序列鎖（SeqLock）的情況下，直接讀取指令區。這在極端高頻下會導致前台讀取到半寫入的髒數據（Dirty Data），引發虛擬機狀態崩潰甚至內存越界。專案對此只做了一般的原子操作保護，並未達到航空級防護標準。這表明為了追求極限的速度，開發者對指令讀寫的內存安全性進行了妥協。

## 三、 no_std 封裝與引用規範性審查
`ScriptGo` 在 `src/lib.rs` 中宣告了 `#![no_std]`，並且在 `Cargo.toml` 中聲明了對 `no_std_tool` 的依賴。

### 3.1 no_std 引用缺失（未實質引用）
- 令人驚訝的是，經代碼全局掃描，`ScriptGo` 的源代碼中**完全沒有引入或使用** `no_std_tool` 的任何 API。
- 它在 `src/lib.rs` 中使用了自定義的暫存器和內存結構，完全沒有調用 `no_std_tool` 提供的同步原語或內存管理。
- 這種「聲明依賴但不使用」的行為，違背了「統一將 no_std 底層封裝在 no_std_tool 中」的規範要求，增加了虛擬機代碼與硬編碼邏輯的耦合度。這說明其 `no_std` 宣告只是停留在編譯器通過層面，未與系統公共組件進行實質融合。

---

## 四、 執行緒生命週期與記憶體釋放安全審查
`ScriptGo` 本身是 `#![no_std]`，在虛擬機運行時內部不創建線程，但在測試與基準測試中頻繁使用線程：
- 在 `benches/concurrency_bench.rs` 和 `examples/tauri_framework/src/main.rs` 中使用了 `thread::spawn`。

### 4.1 基準測試中的執行緒 detached 漏洞
在基準測試中，`writer_handle` 和 `reader_handles` 被創建以測試高頻讀寫，並在結束時調用了 `join()`。然而在 Tauri 示例和 `create-scriptgo-app` 工具中，大量執行緒是在未持有 handle 的情況下被創建的，這導致線程完全被 detached。如果虛擬機在大規模多任務上下文中被銷毀，這些 detached 的執行緒仍可能在背景死循環運行，造成 CPU 熱失控與系統資源洩漏。

## 五、 綜合審計結論與具體改進建議

### 5.1 綜合評級：部分妥協 (Partially Compromised)
虛擬機核心的暫存器執行邏輯非常完整且確實為零分配，但熱更新部分的並發保護不夠嚴密，且未實質引用 `no_std_tool`，與其聲稱的「航天級零妥協」存在差距。

### 5.2 具體改進建議
1. **強制引用 no_std_tool 原語**：
   將虛擬機中的內存庫存儲與多線程狀態同步（如熱更新鎖）替換為 `no_std_tool::sync::SpinMutex`，從而解耦同步邏輯。
2. **引入 SeqLock 保護熱更新**：
   使用序列鎖（SeqLock）來保護虛擬機的指令段更新，在前台執行緒讀取指令時若檢測到背景正在寫入，則進行自旋重試，防止讀取到部分編譯的非法指令。
3. **管理並發線程退出**：
   在所有腳本範例與輔助腳本中，廢除 detached 執行緒，統一使用 `std::thread::Builder` 命名線程，並妥善持有 JoinHandle 以實現主動回收。
